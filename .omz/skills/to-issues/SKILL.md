# To Issues（拆单）

用**垂直切片**（追踪弹）把计划拆成可独立认领的 issue。

问题跟踪器与分诊标签词汇应由项目文档提供；若缺失，使用仓库约定的 setup skill 或询问维护者。

## 流程

### 1. 收集上下文

以对话中已有信息为主。若用户传入工单引用（编号、URL 或路径），从问题跟踪器拉取正文与评论全文。

### 2. 探索代码库（可选）

若尚未了解代码现状，可先探索。issue 标题与正文应使用项目**领域词汇表**用语，并尊重相关区域的 ADR。

### 3. 起草垂直切片

将计划拆成 **追踪弹** issue。每条是一条**薄而贯穿**的竖切：打通**所有**集成层次直至可验证，**不要**按「只做一层」的水平切分。

切片可标为 **HITL** 或 **AFK**。HITL 需要人机协作（如架构拍板、设计评审）。AFK 可由 Agent 独立完成并合并。在可行时**优先 AFK**。

<vertical-slice-rules>
- 每条切片在每一层都打通一条窄但**完整**的路径（如 schema、API、UI、测试——以项目实际分层为准）
- 单条切片完成后即可单独演示或验证
- 宁可多条薄片，不要少量厚块
</vertical-slice-rules>

### 4. 与用户对齐

将拟定拆分以编号列表展示。每条切片包含：

- **标题**：简短描述性名称
- **类型**：HITL / AFK
- **阻塞关系**：须先完成哪些其他切片（若有）
- **覆盖的用户故事**：对应源材料中的哪些故事（若有）

询问用户：

- 粒度是否合适？（过粗 / 过细）
- 依赖关系是否正确？
- 是否需要合并或再拆？
- HITL / AFK 标记是否恰当？

反复迭代直到用户认可拆分结果。

### 5. 发布到问题跟踪器

对每条已认可的切片，在问题跟踪器创建新 issue。正文使用下方模板。这些 issue 默认可交给 AFK Agent，除非另有说明，否则应打上合适的**分诊/就绪**类标签。

按依赖顺序发布（先发布阻塞项），以便在「阻塞于」字段中引用**真实** issue 编号。

## 要做什么

本垂直切片的简明说明。描述**端到端行为**，不要逐层写实现步骤。

避免写死具体文件路径或大段代码——易过时。例外：若原型里有一段比文字更精确地固定了决策（状态机、reducer、schema、类型形状），可摘录在此，并简要注明来自原型；只保留**决策含量高**的片段，不要整份可运行 demo。
