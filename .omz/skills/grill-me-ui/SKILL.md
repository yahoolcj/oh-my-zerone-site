# Grill Me UI

@author: laicaijin
@email:lcj970630@gmail.com
@group:oh-my-zerone

## 目标

`grill-me-ui` 用于 UI 开发前的设计澄清。它复用 `grill-with-docs` 的文档对齐方式，像 `grill-me` 一样逐个追问关键问题，最终输出一份可执行的 UI 设计报告。

本 skill 只做澄清、分析和报告输出，不实现 UI，不创建 PRD/issue，不修改业务代码。

## 边界

### 允许

- 读取 `AGENTS.md`、`docs/agent/introduction.md`、`docs/knowledge/`、`CONTEXT.md`、`CONTEXT-MAP.md`。
- 只读分析用户提供的原型、截图、数据结构、需求描述。
- 只读检索现有页面、组件、样式、设计 token、路由、菜单和相似模块。
- 给出一个主推荐 UI 方案，必要时附带 1-2 个备选方向。
- 输出可交给 `/to-locate` 或 `/to-coding` 的任务描述。
- 发现长期可复用的 UI 规则、业务事实、skill 缺口时，按项目进化机制发起申请。

### 禁止

- 不修改代码。
- 不创建或修改 PRD。
- 不创建或修改 issue。
- 不生成 `.UI/` 原型文件；需要原型文件时应转交 `/to-ui`。
- 不自动更新 `CONTEXT.md`、ADR、知识库、skill、workflow，除非已经按进化机制获得用户确认。
- 不把猜测、不确定内容、未确认术语写成最终结论。

## 工作流

### step1. 设计模式分流

先确认 UI 设计模式：

- `结合现有 UI`：需要与系统已有风格、布局密度、组件规范、样式 token 保持一致。
- `独立设计`：只围绕当前模块设计，不强制兼容全局风格，但必须说明落地边界。

如果用户调用 skill 时已经明确是 `结合现有 UI` 或 `独立设计`，跳过此问题，直接进入对应路径。

推荐问法：

```md
第 1 个问题：
这次 UI 是要结合现有系统风格，还是作为独立模块设计？

我的推荐理解：
如果后续要落到现有项目页面，建议选择“结合现有 UI”；如果只是探索一个新模块或新方向，选择“独立设计”。
```

### step2. 文档与材料对齐

按需只读分析已有信息：

1. `AGENTS.md`
2. `docs/agent/introduction.md`
3. `docs/knowledge/`
4. `CONTEXT.md` / `CONTEXT-MAP.md`
5. 用户提供的原型、截图、数据、需求
6. 现有页面、组件、样式、设计 token

如果某个问题能通过只读分析回答，直接分析，不问用户。

### step3. 一问一答澄清

每次只问一个会影响 UI 决策的关键问题，并给出推荐理解或推荐方案。

优先澄清：

- 目标模块：页面、弹窗、组件、局部区块或完整流程。
- 用户场景：谁在什么场景下使用。
- 内容与数据：需要展示哪些字段、状态、操作。
- 信息层级：哪些内容最重要，哪些可以弱化。
- 交互状态：默认态、加载态、空态、错误态、禁用态、成功反馈。
- 视觉方向：布局、密度、颜色倾向、组件风格。
- 响应式范围：只桌面、只移动端，还是都要。
- 验收口径：后续 `/to-locate` 或 `/to-coding` 如何判断完成。

不要一次性甩完整问卷。用户回答后，继续追问下一个最关键问题。

### step4. 停止追问

当以下信息都明确后，停止追问并输出 UI 设计报告：

1. 设计模式明确。
2. 目标模块明确。
3. 用户场景明确。
4. 内容与数据明确。
5. 信息层级明确。
6. 交互状态明确。
7. 视觉方向明确。
8. 响应式范围明确。
9. 验收口径明确。

如果仍存在影响执行的歧义，继续追问；如果只剩低风险细节，在报告中放入 `待确认项`。

## 进化观察

执行过程中持续观察进化机会，但不得中断主流程。

- 长期有效的业务术语或领域关系：发起 `[进化申请-CONTEXT]` 或 `[进化申请-知识库]`。
- 长期有效的 UI 规范、项目风格规则、skill/workflow 缺口：发起 `[进化申请-技能]`。
- 用户长期偏好或协作规则：按 `docs/agent/evolution.md` 判断是否可写入 `docs/agent/memory.md`。

未获用户确认前，不落盘知识库、CONTEXT、skill 或 workflow。
