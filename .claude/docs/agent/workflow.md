
# Workflow 开发工作流

@group:oh-my-zerone
@version:1.0.0


## fe-dev 前端开发工作流

### step1. [询问]
**should do**： 通过对我进行细节的询问，直到我对所有细节确认为止，保障需求对齐。
**don't**：直接开始改动代码；在用户没有确认前进入其他阶段。
**skills**：通过 **/grill-with-docs** skill进行（如果有这个技能）
**用户确认**：是；
**是否自动确认**：用户允许

### step2. [需求整理] 
**should do**：将上下文讨论结果，PRD文档。
**don't**：直接开始改动代码；在用户没有确认前进入其他阶段。
**skills** 通过 **/to-prd** skill进行生成内容（如果有这个技能）
**用户确认**：是；

### step3. [切片] 
**should do**：将prd文档切片为一个个issue，并创建对应的issue文件记录在.ISSUE/XX下。
**don't**：直接开始改动代码；在用户没有确认前进入其他阶段。
**skills**：通过 **/to-issues** 技能进行切片（如果有这个技能）。
**用户确认**：是；

### step4. [开发]
**should do**：基于指定版本的PRD文档作为上下文需求进行coding开发，开发时必须遵循 **rules/coding.mdc**中的约束规则，一步步完成
**don't**：开发完成后不要进行npm脚本尝试去验证内容，只负责等待用户验收。
**skills**：通过 **/to-coding** 技能进行coding开发（如果有这个技能）。
**用户确认**：自动；


### step5. [测试] 

**should do**：进行issue和PRD TODO测试回收，想用户询问测试情况是否通过，如果不通过继续执行。
**don't**：没有验收通过的issue禁止打勾。
**skills**：
    1.通过 **/to-test** 技能进行测试回收。


### step6. [审查]

**should do**：对本次的改动进行全面审查，并将审查信息反馈给用户，审查遵循 **docs/agent/review.md** 文件中的审查规范。
**skills**：通过 **/to-review** 技能进行审查（如果有这个技能）。


### step7. [提交代码]

**should do**：基于本次任务修改的文件内容，生成commit信息，commit遵循 “commitlint.config” 规范，提交commit后引导或询问用户是否需要提交PR
**skills**：通过 **/to-commit** 技能进行提交commit信息（如果有这个技能）。





## fe-locate 前端问题定位工作流

用于已开发功能中的小问题、小需求、BUG、交互问题、样式问题、数据异常等轻量定位场景。该工作流只定位，不开发；最终产物作为 `/to-coding` 的输入。

### step1. [术语与问题澄清]

**should do**：
- 读取 `AGENTS.md`、`docs/agent/introduction.md`、`docs/knowledge/`、`CONTEXT.md` / `CONTEXT-MAP.md` 等已有信息。
- 对不确定业务词汇、页面名、模块名、行为预期进行逐项澄清。
- 每次只问一个关键问题，并给出“我的推荐理解”。
- 对 BUG 补齐页面/入口、操作步骤、当前表现、期望表现、影响范围、是否稳定复现。
- 对小需求补齐页面/入口、调整对象、目标行为/文案/样式、生效范围、验收口径。
- 持续观察术语、知识、skill、workflow 是否触发进化申请。

**don't**：
- 不改代码。
- 不创建 PRD / issue。
- 不把猜测、不确定内容、未确认术语作为最终结论。
- 未获用户确认前，不落盘知识库、CONTEXT、skill 或 workflow。

**skills**：通过 **/to-locate** 技能进行。
**用户确认**：是。

### step2. [只读定位]

**should do**：
- 通过只读方式检索路由、菜单、页面、组件、状态、接口、样式、类型定义。
- 默认按“文档/知识 → 路由/菜单 → 页面 → 组件 → 状态/接口 → 样式”检索。
- 按问题类型调整优先级：文案/样式优先页面和组件；数据异常优先 API、store、hooks；权限/不可见优先路由、菜单、权限；交互异常优先组件、事件、状态流。
- 定位到具体文件路径和符号名，包含组件名、函数名、变量名、store 名、接口名、路由名、样式类名或 token 名。
- 如果出现新不确定点，回到 step1 继续澄清。

**don't**：
- 不做实现。
- 不运行会改变工作区状态的命令。
- 不输出“可能”“大概”作为最终结论。
- 不主动用定位失败报告结束流程；除非用户明确终止，否则继续追问和检索。

### step3. [定位报告]

**should do**：
- 输出最终定位报告或轻量定位报告。
- 报告必须包含确认后的需求、问题类型、代码定位、证据、影响范围、可交给 `/to-coding` 的任务描述。
- 报告必须说明流程边界：未修改代码、未创建 PRD/issue、未提交 commit。
- 如果有进化机会，在报告末尾统一附带进化申请。

**don't**：
- 不保留未确认的不确定点。
- 不进入 `/to-coding`，除非用户明确要求。
- 不修改任何文件，除非用户对进化申请给出确认授权。




## fe-explain 项目解释工作流

用于用户不了解项目中的代码、业务概念、模块、流程、字段、权限规则或关系图时，提供只读学习理解说明书。该工作流是项目“翻译官”，只解释，不定位修复，不开发，不创建 PRD/issue。

### step1. [解释对象澄清]

**should do**：
- 识别解释对象类型：代码对象或业务对象。
- 代码对象包含文件、组件、函数、hook、store、接口、路由、样式、配置等。
- 业务对象包含业务术语、页面功能、用户流程、领域模型、权限规则、数据含义等。
- 如果用户输入模糊，每次只问一个关键问题，并给出“我的推荐理解”。
- 如果用户要求“先按你的理解解释”，可以继续，但必须在报告中标注假设。
- 持续观察术语、知识、skill、workflow、文档是否触发进化申请。

**don't**：
- 不修改代码。
- 不创建 PRD / issue。
- 不运行项目脚本、构建、测试或 dev server。
- 不把猜测、不确定内容、未确认术语作为最终结论。
- 未获用户确认前，不落盘知识库、CONTEXT、skill、rule、workflow 或项目文档。

**skills**：通过 **/to-explain** 技能进行。
**用户确认**：有关键歧义时需要。

### step2. [只读信息收集]

**should do**：
- 按优先级读取 `AGENTS.md`、`docs/agent/workflow.md`、`docs/agent/permission.md`、`docs/agent/introduction.md`、`CONTEXT.md`、`CONTEXT-MAP.md`、`docs/knowledge/`、`docs/agent/memory.md`。
- 通过只读方式检索路由、菜单、页面、组件、hooks、store、API、types、styles、测试、mock、配置等证据。
- 对核心结论绑定证据来源，包含文件路径、符号名、文档路径或检索证据。
- 如果文档与代码冲突，以当前代码作为当前行为证据，同时标记文档疑似过期，并在报告末尾提出进化申请。

**don't**：
- 不做实现。
- 不运行会改变工作区状态的命令。
- 不输出“可能”“大概”作为最终结论。
- 不把证据不足的业务规则画进关系图当事实。

### step3. [解释报告]

**should do**：
- 输出学习理解说明书，默认中文，代码标识、文件路径、函数名、变量名和接口字段保持原文。
- 按用户需要选择快速解释、标准解释或深度解释；用户未指定时默认标准解释。
- 报告包含解释对象、一句话结论、背景与用途、证据来源、关键入口、Mermaid 关系图、执行/交互流程、关键概念表、事实/推断/待确认、容易误解点、建议阅读顺序、可继续追问的问题、进化观察。
- 宽泛问题先输出分层解释地图，再展开核心链路。
- 关系图按对象类型选择 flowchart、graph、sequenceDiagram、stateDiagram-v2 或 erDiagram。
- 报告必须说明流程边界：未修改代码、未创建 PRD/issue、未提交 commit。

**don't**：
- 不保留未确认的不确定点作为事实。
- 不进入 `/to-locate` 或 `/to-coding`，除非用户明确要求。
- 不修改任何文件，除非用户对进化申请给出确认授权。
