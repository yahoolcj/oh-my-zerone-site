# 问题定位（to-locate）

## 目标

`/to-locate` 用于处理已开发功能中的小问题、小需求、BUG、交互问题、样式问题、数据异常等轻量定位场景。

本 skill 只产出问题定位报告，作为后续 `/to-coding` 的明确上下文输入。

## 边界

### 允许

- 读取 `AGENTS.md`。
- 读取 `docs/agent/*`。
- 读取 `docs/knowledge/*`。
- 读取 `CONTEXT.md`、`CONTEXT-MAP.md` 或上下文目录中的 `CONTEXT.md`。
- 只读检索路由、菜单、页面、组件、状态、接口、样式、类型定义等代码。
- 使用只读命令，如 `rg`、`Get-Content`、`git grep`、`git log`、`git diff --name-only`。
- 输出定位假设、澄清问题、最终定位报告、轻量定位报告。
- 在最终报告末尾统一提出进化申请。

### 禁止

- 不修改代码。
- 不创建或修改 PRD。
- 不创建或修改 issue。
- 不创建或修改知识库、CONTEXT、skill、workflow，除非已经按进化机制获得用户确认。
- 不运行会改变工作区状态的命令。
- 不提交 commit。
- 不直接进入 `/to-coding`，除非用户明确要求。
- 不把猜测、不确定内容、未确认术语作为最终结论。

## 工作流

### step1. 术语与问题澄清

先理解用户描述中的业务词、页面名、模块名、行为预期。

优先查证：

1. `AGENTS.md`
2. `docs/agent/introduction.md`
3. `docs/knowledge/`
4. `CONTEXT.md` / `CONTEXT-MAP.md`
5. 路由、菜单、页面目录、模块命名、组件命名

如果仍无法确定，必须向用户追问。每次只问一个关键问题，并给出推荐理解：

```md
我还不能确定：...
我的推荐理解：...
请确认：...
```

术语澄清规则：

- 业务概念适合沉淀到 `CONTEXT`。
- 实现定位知识、路径映射、项目事实适合沉淀到 `docs/knowledge/`。
- 沉淀前必须按 `docs/agent/evolution.md` 发起进化申请，未确认前禁止落盘。

### step2. 最小信息补齐

BUG 至少明确：

- 页面/入口
- 操作步骤
- 当前表现
- 期望表现
- 影响范围
- 是否稳定复现

小需求至少明确：

- 页面/入口
- 要调整的对象
- 目标行为、文案或样式
- 生效范围
- 验收口径

缺失信息时逐项追问，不一次性甩表单。

### step3. 只读定位

默认检索顺序：

1. 项目语义层：`AGENTS.md`、`docs/agent/introduction.md`、`docs/knowledge/`、`CONTEXT.md`。
2. 入口层：路由表、菜单配置、页面目录、权限、微前端入口。
3. 页面层：page、layout、page model、hooks。
4. 组件层：业务组件、通用组件、表单、弹窗、卡片、列表。
5. 数据与状态层：store、hooks、API、query、mutation。
6. 表现层：CSS、LESS、SCSS、Tailwind、className、theme、token。

可按问题类型调整优先级：

- 文案/样式：优先页面和组件。
- 数据异常：优先 API、store、hooks。
- 权限/不可见：优先路由、菜单、权限。
- 交互异常：优先组件、事件、状态流。

定位报告的代码证据必须尽量具体到：

- 文件路径
- 组件名
- 函数名
- 变量名
- store 名
- 接口名
- 路由名
- 样式类名或 token 名

如果只能定位到目录，继续追问或继续只读检索，直到能缩小到具体文件和符号。

### step4. 深度澄清闭环

如果信息不足或定位失败，不主动输出失败报告作为结束态。

必须像 `grill-me` 一样继续深挖：

- 不结束。
- 不用“可能”“大概”作为最终结论。
- 继续问用户一个最关键问题。
- 每个问题都给推荐答案。
- 用户回答后继续交叉验证代码和文档。

直到满足：

1. 术语明确。
2. 页面/模块明确。
3. 预期行为明确。
4. 影响范围明确。
5. 代码位置明确。
6. 可交给 `/to-coding`。

只有用户明确说“停止”“不用定位了”“先到这里”，才允许输出阶段性中断摘要。

### step5. 输出最终定位报告

满足以下条件时，可以直接输出最终报告，不再追问：

- 用户问题没有未定义业务术语。
- 页面、模块、功能指向明确。
- 预期行为明确。
- 已定位到具体文件和符号。
- 没有影响 `/to-coding` 的歧义。
- 没有需要用户选择的实现方向。

小问题允许输出轻量定位报告，但不能跳过澄清规则。

## 输出模板

### 标准定位报告

```md
## 问题定位报告

### 用户问题
...

### 问题类型
BUG / 小需求 / 样式 / 交互 / 数据

### 确认后的需求
...

### 定位结论
...

### 代码定位
- `path/to/file.tsx`
  - 符号：`ComponentOrFunction`
  - 证据：...
  - 相关性：高 / 中 / 低

### 影响范围
...

### 建议进入 /to-coding 的任务描述
- 修改目标：
- 涉及文件：
- 预期行为：
- 不应改动：
- 验收口径：

### 流程边界
本次 `/to-locate` 仅完成问题定位：
- 未修改代码
- 未创建 PRD / issue
- 未提交 commit
- 可作为 `/to-coding` 输入

### 进化申请
如有长期有效术语、项目事实或 skill/workflow 更新机会，在这里统一附带申请。
```

### 轻量定位报告

```md
## 轻量定位报告

问题类型：文案/样式/交互/小需求/BUG
确认后的需求：...
相关文件：...
代码证据：...
/to-coding 任务描述：...
流程边界：本次只定位，未修改代码、未创建 PRD/issue、未提交 commit。
进化申请：如有则附带。
```

## 进化观察

定位过程中持续观察进化机会，但不中断定位主流程。

最终报告末尾统一提出申请：

- 业务领域术语：`[进化申请-CONTEXT]`
- 项目事实、路径映射、实现知识：`[进化申请-知识库]`
- skill、workflow、长期协作规则：`[进化申请-技能]`

未获用户确认前，禁止落盘。
