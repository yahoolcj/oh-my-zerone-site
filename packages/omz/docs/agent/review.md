## Output Notes（输出规范）

Review 报告应使用 Markdown 分块内容，不使用表格展示检查项。

每个检查项至少包含：

- 状态：`通过` / `未通过` / `待确认`
- 证据：列出文件路径和判断依据
- 问题：未发现时写 `未发现`
- 建议：无建议时写 `无`
- 结论：说明该项是否完成到位

最终结果规则：

- 任一检查项为 `未通过`，最终结果为 `未通过`。
- 没有 `未通过`，但存在 `待确认`，最终结果为 `待确认`。
- 所有检查项均为 `通过`，最终结果为 `通过`。

## Task Checklist Template（本次 Review 清单模板）

当用户需要针对某个功能、组件、PRD、issue 或修复结果提供专项 Review 清单时，使用以下格式：

```markdown
本次 review 清单：
- 检查项 A
- 检查项 B
```

只有标题行之后紧邻的连续 `- ` 列表会被视为正式本次 Review 清单。普通描述、关注点、担忧或补充说明只作为上下文，不自动视为检查项。

## Re-review Notes（复检说明）

复检时应读取用户贴出的上次 Review 报告，或读取用户指定的 Review 报告路径。

复检优先检查上次报告中的 `未通过` 和 `待确认` 项，并说明每项当前状态：

- `已关闭`：已有证据证明问题已修复或确认通过。
- `仍未通过`：问题仍存在。
- `仍待确认`：仍需要用户或人工体验确认。

如果用户没有提供上次报告，则执行一次新的完整 Review，不声称是在关闭上次问题。
