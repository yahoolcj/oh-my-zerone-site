# 代码质量质检（to-quality-review）

## 目标

`/to-quality-review` 用于对指定代码范围进行静态代码质量质检，输出建议型质检报告。

本 skill 只提出代码质量建议，不输出 `通过` / `不通过` 结论，不作为交付门禁。硬性要求、验收规则、PRD 符合度、复检上次未通过项由 `/to-review` 处理。

## 硬边界

- 不修改代码、文档、PRD、issue、配置或生成文件。
- 不运行项目脚本，包括但不限于 `npm run lint`、`npm run build`、`npm test`、`pnpm test`。
- 不提交、不暂存、不推送、不创建 PR。
- 不默认落盘质检报告；只有用户明确要求保存或归档时才允许写入文件。
- 不把建议型问题表述成硬性阻塞。
- 不忽略项目已确认的上下文决策。
- 不输出大段完整重写实现，避免从质检越界成开发。

## 审查范围

按以下优先级确定审查范围：

1. 用户明确指定的文件、目录、组件、PRD、issue 或功能范围。
2. 用户未指定时，审查当前 `git diff` 涉及的代码文件。
3. 如果用户未指定范围且当前无 diff，停止并询问用户要质检的范围。

只审查代码质量相关文件。明显无关的锁文件、生成文件、构建产物、图片等默认排除，除非用户明确指定。

## 上下文读取

审查前必须读取项目上下文，避免反复警告用户已经确认的决策。

优先读取：

1. `CONTEXT.md`
2. `CONTEXT-MAP.md`
3. `docs/knowledge/`
4. `docs/agent/introduction.md`
5. `rules/coding.mdc`
6. 相关目录附近的 README、上下文说明、既有相似实现

如果某个写法已有明确上下文决策支持，不允许作为普通问题输出；只能输出为 `上下文豁免` 或残留风险提示。

如果上下文文件之间存在冲突，标记为 `未确认`，不要自行选择一边。

如果上下文缺失，才按通用代码质量标准判断。

## 检查维度

默认检查以下维度，用户可以在本次请求中追加专项关注点：

- 命名规范：函数、变量、组件、类型、文件命名是否表达准确。
- 过度封装：是否为了复用制造不必要抽象、参数透传、层级过深。
- 函数职责：单个函数或组件是否职责过多、分支过密、难测试。
- 边界处理：空值、异常状态、异步失败、数组为空、权限缺失、类型不匹配。
- 性能风险：重复计算、无意义重渲染、大列表处理、昂贵逻辑放在渲染路径。
- 可读性：条件判断、数据转换、命名语义、局部变量拆分是否清晰。
- 可维护性：重复逻辑、隐式耦合、魔法值、跨层调用、改动影响面。
- 类型质量：类型是否过宽、`any` 滥用、断言逃避、接口定义不稳定。
- 项目规范贴合度：是否符合现有目录职责、已有工具函数、局部代码风格。
- 优化空间：不一定是问题，但有更简单、更稳定或更清晰的写法。

## 建议分级

报告只使用以下建议分级：

- `强烈建议修改`：性能风险、明显边界缺失、可能造成线上异常、复杂度明显失控、后续维护成本显著增加。
- `建议修改`：命名不清、职责偏大、重复逻辑、轻度过度封装、类型定义偏宽、可读性较差。
- `可选优化`：有更优写法，但当前实现可接受，主要提升长期可读性或一致性。
- `上下文豁免`：通用最佳实践会提醒，但项目文档、CONTEXT、知识库或既有决策明确支持当前写法。
- `未确认`：需要用户补充上下文，或仅凭静态阅读无法判断是否真有问题。

性能类问题默认至少标为 `强烈建议修改`，除非项目上下文明确接受该性能取舍。

## 上下文豁免规则

当通用代码质量建议与项目已确认决策冲突时，项目已确认决策优先。

报告中应写为：

- `级别：上下文豁免`
- 说明上下文来源。
- 说明残留风险。
- 给出低压建议，例如“后续若重构可考虑”。

不得把已确认决策反复作为普通问题警告。

## 触及框架或底层的建议

如果建议改法触及框架、底层、公共能力或跨模块契约，必须主动标记需要方案确认，并建议用户先进入 `/grill-me`。

包括但不限于：

- 框架核心逻辑。
- 底层工具层。
- 公共组件。
- 全局 store、请求封装、路由守卫、权限体系。
- 跨模块数据结构或接口契约。
- 多页面共享逻辑。
- 会导致多个模块联动修改的性能优化。

报告必须额外说明：

- 为什么会影响其他模块。
- 可能影响哪些模块。
- 直接进入 `/to-coding` 的风险。
- 建议先讨论哪些方案点。

## 改法建议边界

报告允许包含代码级改法方向，但不能直接修改代码。

- 对命名问题：给出推荐命名。
- 对过度封装：指出可以删除、合并或下沉的抽象层。
- 对边界处理：列出应补的边界条件。
- 对性能问题：说明优化方向，例如缓存、提前计算、减少渲染路径计算、避免重复创建对象或函数。
- 对局部小改动：可以给出可交给 `/to-coding` 的修复任务。
- 对触及框架或底层的改动：先建议 `/grill-me`，不要直接包装成普通 `/to-coding` 任务。

## 执行流程

1. 读取用户请求，识别专项关注点。
2. 按范围规则确定审查范围。
3. 读取上下文来源，记录会影响判断的既有决策。
4. 只读检索和阅读审查范围内的相关代码。
5. 按检查维度逐项判断。
6. 对每个发现给出分级、位置、证据、影响、建议和上下文判断。
7. 输出 Markdown 质检报告。
8. 只有用户明确要求保存或归档时，才写入 `.QUALITY_REVIEW/<yyyy-mm-dd>-<topic>-quality-review.md` 或用户指定路径。

## 结果规则

- 不输出最终 `通过` / `不通过`。
- 不使用表格展示发现项。
- 所有发现必须有具体文件路径、符号或代码证据。
- 没有发现问题时，也要说明已检查维度和边界。
- 静态阅读无法确认的内容标为 `未确认`，不得假装已运行验证。
