# CONTEXT.md 格式

## 结构

```md
# {上下文名称}

{一到两句话描述该上下文是什么，以及为什么存在。}

## 语言术语

**订单（Order）**：
{对该术语的简洁描述}
_避免使用_：购买、交易

**发票（Invoice）**：
交货后向客户发出的付款请求。
_避免使用_：账单、付款请求

**客户（Customer）**：
下单的个人或组织。
_避免使用_：客户端、买家、账户

## 关系

- 一个 **订单** 产生一张或多张 **发票**
- 一张 **发票** 属于且仅属于一个 **客户**

## 示例对话

> **开发者：** "当 **客户** 下 **订单** 时，我们立即创建 **发票** 吗？"
> **领域专家：** "不——**发票** 只有在 **履约** 确认后才会生成。"

## 已标记的歧义

- "账户"曾同时指代 **客户** 和 **用户** ——已解决：这是两个不同的概念。
```

## 规则

- **要有主见。** 当同一概念存在多种表达时，选择最合适的一个，并将其余列为应避免使用的别名。
- **明确标记冲突。** 如果某个术语被歧义使用，在"已标记的歧义"中指出，并给出明确的解决方案。
- **保持定义简洁。** 最多一句话。定义它是什么，而不是它做什么。
- **展示关系。** 使用加粗术语名，并在明显时表达基数关系。
- **只收录本项目上下文特有的术语。** 通用编程概念（超时、错误类型、工具模式）即使在项目中大量使用也不属于这里。添加术语前先问：这是本上下文特有的概念，还是通用编程概念？只有前者才应收录。
- **当自然形成聚类时，使用子标题对术语分组。** 如果所有术语都属于同一个内聚领域，平铺列表即可。
- **撰写示例对话。** 一段开发者与领域专家之间的对话，自然地展示术语的交互方式，并澄清相关概念之间的边界。

## 单上下文与多上下文仓库

**单上下文（大多数仓库）：** 在仓库根目录放一个 `CONTEXT.md`。

**多上下文：** 在仓库根目录放一个 `CONTEXT-MAP.md`，列出各上下文、它们的位置以及相互关系：

```md
# 上下文地图

## 上下文列表

- [订单（Ordering）](./src/ordering/CONTEXT.md) —— 接收并跟踪客户订单
- [账单（Billing）](./src/billing/CONTEXT.md) —— 生成发票并处理付款
- [履约（Fulfillment）](./src/fulfillment/CONTEXT.md) —— 管理仓库拣货和发货

## 关系

- **订单 → 履约**：订单发出 `OrderPlaced` 事件；履约消费这些事件来启动拣货
- **履约 → 账单**：履约发出 `ShipmentDispatched` 事件；账单消费这些事件来生成发票
- **订单 ↔ 账单**：共享 `CustomerId` 和 `Money` 类型
```

该 skill 的推断规则：

- 如果存在 `CONTEXT-MAP.md`，读取它来查找各上下文
- 如果只存在根目录的 `CONTEXT.md`，则为单上下文
- 如果两者都不存在，在第一个术语确定时按需创建根目录的 `CONTEXT.md`

当存在多个上下文时，推断当前话题属于哪个上下文。如果不确定，直接询问。
