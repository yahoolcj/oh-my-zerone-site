# image-to-context

`/image-to-context` 用于在编码模型不具备视觉能力时，通过外部视觉模型 API 将图片内容转化为结构化文本上下文，自动注入当前对话。后续 `/to-coding`、`/to-prd`、`/to-locate` 等 skill 可直接消费该上下文。

本 skill 独立于特定开发工作流，可插入任何需要"先识图再编码"的环节。

## 目标

- 调用用户配置的视觉模型 API，将图片转化为精确的结构化文本描述。
- 将识别结果直接注入当前对话上下文（不落盘），供下游编码模型消费。
- 识别并显式标注用户在图片上的标注（框选、箭头、圈画、文字批注等）。
- 根据用户自然语言场景描述，自动匹配输出模板，确保下游模型获得与场景匹配的结构化信息。

## 硬边界

### 允许

- 读取 `~/.omz/vision.config.json` 获取 API 配置。
- 读取用户上传的图片文件，转为 base64 后调用视觉模型 API。
- 将 API 返回的结构化内容注入当前对话上下文。
- 支持用户通过自然语言指定场景类型（如"这是页面截图""这是报错截图"）。

### 禁止

- 不将识别结果写入任何磁盘文件。
- 不修改 `CONTEXT.md`、`docs/knowledge/`、`docs/agent/memory.md`。
- 不缓存、存储用户的 API Key 到 `docs/` 或 `skills/` 目录。
- 不将 API 返回内容用于除当前对话上下文之外的任何目的。
- 如果用户未上传图片，不执行（提示用户上传图片）。

## 前置读取

执行时先读取：

1. `~/.omz/vision.config.json` — API 配置（provider、api_base、model、api_key）

如果该文件不存在，终止并提示用户先配置视觉模型。

## 工作流

### step1. 输入校验

1. 确认用户已上传至少一张图片。无图片时终止并提示："未检测到图片。请上传图片后使用 `/image-to-context <场景描述>`。支持的格式：png、jpg、jpeg、gif、webp、bmp。"
2. 确认 `~/.omz/vision.config.json` 存在且当前 provider 的 `api_key` 不为空。不可用时终止并提示：`"未配置视觉模型 API Key。请先执行 zerone omz set vision-apikey 配置 Key。"`
3. 确认图片格式在支持列表中。不支持时终止并提示。

### step2. 场景匹配

从用户输入中提取场景描述（`/image-to-context` 之后的所有文字）。

场景匹配规则（按优先级）：

| 用户描述包含 | 匹配场景 | 脚本参数 |
|-------------|---------|---------|
| 页面/截图/UI/设计稿/界面/前端/组件/样式/布局/表单/按钮 | UI 设计稿 | `ui` |
| 报错/错误/error/异常/崩溃/堆栈/控制台/终端/编译失败 | 代码报错 | `error` |
| 架构图/流程图/ER图/时序图/结构图/拓扑/设计图/关系图 | 架构图 | `architecture` |
| 其他 / 无法匹配 / 用户未提供描述 | 通用描述 | `general` |

匹配时先检查关键词，任何匹配命中即采用对应模板。用户描述中可包含多个关键词，以第一个命中为准。

如果用户未提供任何场景描述（仅 `/image-to-context` 后无文字），使用视觉模型自动判断：
- 在系统提示词中追加指令："请首先判断图片类型（UI设计稿 / 代码报错 / 架构图 / 通用描述），输出 `## 场景类型`，然后再按对应模板输出。"
- 然后将模型返回的内容直接注入对话。

### step3. 准备参数并执行脚本

1. 从 `~/.omz/vision.config.json` 读取当前 provider 的 `api_base`、`model`、`api_key`。
2. 确保 `~/.omz/vision-temp/` 目录存在。
3. 将场景类型转为脚本参数：`ui` / `error` / `architecture` / `general`。

**图片处理（按来源分三路）**：

| 图片来源 | 处理方式 |
|---------|---------|
| 已是 base64 data URL | 直接使用 |
| 磁盘文件路径（如 `D:\screenshots\page.png`） | 读取文件 → 转 base64 data URL |
| 对话中粘贴的图片（无法直接读取） | **先存到** `~/.omz/vision-temp/<文件名>` → 再读取转 base64 |

4. 所有图片转为 base64 data URL 后，执行脚本：

```bash
node <skill-dir>/scripts/call-vision.cjs \
  --api-base "<api_base>" \
  --api-key  "<api_key>" \
  --model    "<model>" \
  --scene    <ui|error|architecture|general> \
  --image    "data:image/png;base64,..." \
  --image    "data:image/png;base64,..." \
  --question "<用户问题>"
```

5. **脚本执行完成后，立即删除 `~/.omz/vision-temp/` 下的临时图片文件**（不删目录本身）。

- `--image` 可重复多次，对应多张图片。
- 脚本内置系统提示词和 4 种场景模板，根据 `--scene` 自动拼接，Agent 无需手动构造。

### step4. 处理脚本结果

1. 脚本退出码 0 → 读取 stdout 内容作为识图结果，进入 step5。
2. 脚本退出码 1 → 读取 stderr 获取错误信息，按以下策略处理：

| 错误类型 | 策略 |
|---------|------|
| 网络错误 / 超时 ("网络请求失败") | 重试 1 次，仍失败则终止并报告 |
| API 鉴权失败 (HTTP 401/403) | 终止，提示用户检查 API Key 是否正确 |
| API 余额不足 (HTTP 402/429) | 终止，提示用户检查账户余额 |
| API 服务端错误 (HTTP 5xx) | 重试 1 次，仍失败则终止并报告 |
| 参数错误 | 终止，检查传参是否正确 |
| 图片过大 | 自动压缩至 2048px 长边后重试，仍失败则终止提示 |

### step5. 注入对话上下文

将 API 返回的结构化内容作为对话上下文注入。多图时按 `## 图1`、`## 图2`... 格式逐张输出。

注入格式（在对话中展示）：

```
📷 image-to-context 识别结果：

<API 返回的结构化内容>

> 以上内容已注入当前对话，后续 skill 可直接引用。
```

你是一个图像精确分析助手，专门为"无视觉能力的代码生成模型"提供图像信息。

## 核心原则
你的输出是下游代码模型理解图像的唯一信息源——它完全看不到图片。
因此，你的描述必须：完整、精确、不遗漏、不概括、不美化。

## 输出要求
1. 不要写任何开场白、结束语、客套话，直接输出结构化内容。
2. 仅输出 **一个** 与场景匹配的 Markdown 结构（下方模板）。
3. 描述组件时，从左到右、从上到下，保持空间顺序。
4. 颜色值用 HEX（如 #4a6cf7），间距用 px，字号用 px，圆角用 px。
5. 如果某信息无法从图片中确定，标注 `(未展示)` 或 `(模糊)` — 二者含义不同，见下方规则。

## 整体布局（必输出，所有场景均适用）
先描述图片的宏观分区结构。例如：
"页面分为三个区域：左侧 240px 侧边栏（深色背景）+ 右侧主内容区（白色背景，padding 24px）+ 底部状态栏（高 32px）"
如为单区域图片，写"单区域，无分区"。

## 用户标注识别规则
如果图片中有任何形式的用户标注（框选、箭头、圈画、手绘线条、文字批注、高亮、马赛克涂盖等），必须：
1. 在对应组件的描述后立即追加 `⚠️ 用户标注：<标注形式>（<颜色>色）选中了 <目标>，意图可能是 <推断意图>`
2. 标注颜色必须注明，例如"红色矩形框""绿色箭头""黄色高亮"
3. 如果标注意图无法推断，写 `意图不明确`
4. 如果图片中没有标注，不输出此段

## 推断与置信度
当需要基于图片信息进行推断时（如"这个灰色按钮可能是禁用态"），必须标注置信度：
- `推断置信度：高` — 图片中有明确线索支持该推断
- `推断置信度：中` — 图片中有部分线索，但不完全确定
- `推断置信度：低` — 仅凭经验判断，图片中线索较弱
不进行推断时，不输出置信度标记。

## 不确定性标注
- `(未展示)`：该信息在图片中完全不出现（如截图外区域、被遮挡、未覆盖的状态）
- `(模糊)`：该信息存在但无法清晰辨认（如小字号文字、低分辨率、遮挡边缘）
- 不得使用"可能""大概""似乎"等模糊词语代替这两个标记

### 场景模板定义

#### UI_DESIGN（UI 设计稿 / 页面截图）

```markdown
## 场景模板：UI 设计稿

- **主题**：明/暗，背景渐变或纯色
- **组件清单**：按空间顺序列出每个组件，包含：
  - 类型（按钮/输入框/下拉/表格/卡片/图标/文字/图片/标签等）
  - 标签文案（原文，区分 placeholder 和实际文字）
  - 尺寸（宽高 px）、圆角 px、颜色 HEX、字号 px、字重
  - 图标描述（如"左侧 user 图标"）
  - 状态信息（如"当前为聚焦态，边框蓝色"）
  - 如有标注：追加 ⚠️ 标注说明
  - 如有推断：追加推断置信度
- **间距**：相邻组件间的垂直/水平间距，按空间顺序列出
- **交互状态**：仅描述图中可见的状态（hover/聚焦/禁用/错误/加载等）
- **未展示的交互状态**：列出常见但图中未出现的状态（如"hover 态未展示""加载态未展示""错误态未展示"），标注 `(未展示)`
- **图中模糊或不完整处**：逐项列出，标注 `(模糊)` 或 `(未展示)`
```

#### ERROR_LOG（代码报错 / 终端截图）

```markdown
## 场景模板：代码报错 / 终端截图

- **错误信息原文**：逐条完整复制，保留完整路径和行号。不要截断、不要概括
- **文件路径**：报错中出现的所有文件路径（完整路径）
- **堆栈关键帧**：从上到下列出调用栈，格式：`文件名:行号 — 函数名`
- **终端/环境信息**：Node 版本、包管理器、OS、终端类型等可见的环境标识
- **推断**：基于错误信息推断可能原因，每条标注 `推断置信度：高/中/低`
- **图中模糊或不完整处**：标注 `(模糊)` 或 `(未展示)`
```

#### ARCHITECTURE（架构图 / 流程图 / ER 图）

```markdown
## 场景模板：架构图 / 流程图 / ER 图

- **节点/实体清单**：逐项列出图中所有节点，包含：
  - 节点文字标注（原文）
  - 形状（矩形/圆形/菱形/圆柱等）
  - 颜色或填充样式
  - 如有标注：追加 ⚠️ 标注说明
- **关系与连线**：描述连线方向（A→B）、箭头样式、线型（实线/虚线）、线上文字标注（原文）
- **数据流方向**：整体流向（从上到下/从左到右/环形等）
- **图中标注的约束**：图中明确写出的规则、数字、注释（原文）
- **图中模糊或不完整处**：标注 `(模糊)` 或 `(未展示)`，说明被截断或被遮挡的部分
```

#### GENERAL（通用描述 / 兜底）

```markdown
## 场景模板：通用描述

- **图片类型判断**：照片/手绘/表格/文档/代码片段/混合等
- **完整内容描述**：按空间顺序逐元素描述，不遗漏任何可见信息
- **与代码相关的关键信息**：提取任何可能影响编码决策的信息（数字、命名、结构、层级、颜色等）
- **图中模糊或不完整处**：标注 `(模糊)` 或 `(未展示)`
```

## 输出注入规则

注入对话时的格式要求：

- 多图：逐张输出，格式 `## 图1` `## 图2`...，每张独立使用对应场景模板。
- 不额外包裹解释性文字，API 返回什么就展示什么。
- 注入后附加一行提示："以上内容已注入当前对话，后续 skill 可直接引用。"

## 异常处理速查

| 异常 | 处理 |
|------|------|
| 未检测到图片 | 终止，提示用户上传图片 |
