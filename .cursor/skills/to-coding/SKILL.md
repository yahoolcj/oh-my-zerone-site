# PRD+Issue驱动编码（to-coding）

## 范围

- **需求来源**： 依据用户提供的TODO内容或指定的PRD版本文件，用户提供或对话中明确的版本号文件名优先；若未指定，询问用户要使用哪一个 PRD 文件。
- **引导/to-prd**：若用户的需求不是清晰的TODO清单，引导用户先通过/to-prd进行需求整理，生成清晰的TODO清单。然后再进行开发。
- **TODOQ**：PRD 中用于逐项完成的勾选清单，通常出现在 **「功能清单」**（或同等语义的章节）下，格式为 `- [ ]` / `- [x]`。若 PRD 使用其他标题但语义相同，仍视为 TODOQ。
- **唯一执行原则**：严格按照PRD文档执行，不做PRD之外的内容，不在没有对应PRD的情况下工作。


## 硬性规则

1. **顺序执行**：按 TODOQ 从上到下，**同一时间只聚焦一条未完成项**。实现并自测通过后，向用户说明本项做了什么、如何验证，并 **停止推进下一项**，等待验收。
2. **用户验收门禁**：只有用户明示该项验收通过（或确认可勾选）后，才允许将 PRD 中对应行的 `- [ ]` 更新为 `- [x]`。**禁止**在用户未确认前擅自勾选或批量勾选。
3. **及时提交代码**：在完成每项TODO后，若用户确认验收通过，则引导用户使用 **/to-commit** 技能进行提交代码。
4. **PRD 文件**：**禁止**主动修改 PRD 的正文（概述、数据模型、交互说明等）。**允许**的唯一 PRD 变更是：在用户验收某条 TODOQ 后，将 **该条** 的复选框从未完成改为已完成；不得重排清单、不得删减或扩写需求段落。
5. **例外**：若用户**明确要求**修订 PRD（例如修正笔误、对齐已达成共识的范围），须在修改前取得用户确认，并在回复中说明改了何处、为何；仍不推荐在编码过程中顺手改需求文。



## 每项工作流（单条 TODOQ）

```
1. 读取当前 PRD，定位第一条仍为 - [ ] 的 TODOQ
2. 结合 PRD 全文（概述、模型、交互、约束）实现本项，遵守 AGENTS.md 与 .cursor/rules/coding.mdc
3. 按 AGENTS.md「Verification」运行相应命令（如 npm run build:packages）
4. 回复用户：本项交付摘要、验证方式、剩余未完成项列表
5. 等待用户验收
6. 用户确认后：仅更新该条 - [ ] → - [x]，然后回到步骤 1
```

## 实现约定（本仓库）

- 开发过程严格遵循 rules/coding.mdc 中的要求。
- 新增/修改代码遵循项目组件目录、导出、`z1-*` 命名等约定
- 不新增依赖，除非 PRD 或用户明确要求。
- 保持 diff 小、可回退；不做与当前 TODOQ 无关的重构。




## 状态沟通模板（每次完成一项后）

向用户提供：

- **当前 PRD 文件**：`.PRD/<版本>.md`
- **刚完成**：TODOQ 第 N 条（引用原文一行）
- **改动要点**：涉及路径与行为
- **如何验证**：已执行的命令 + 建议的手动检查（若有）
- **待验收**：请用户确认本条是否通过；通过后你再将 PRD 中该条勾选为完成。


## 不需要做的事

- 在开发过程中，你不需要去执行npm run dev 或npm run build等package.json中的脚本，这些交由用户自行执行，你只要继续保持开发顺利进行。



若某条 TODOQ 在现有代码中已明显满足，仍须向用户说明依据并请其确认后，方可勾选。
