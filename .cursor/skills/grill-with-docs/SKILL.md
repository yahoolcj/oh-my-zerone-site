## 领域感知

在探索代码库时，同时查找已有的文档：

### 文件结构

大多数仓库只有一个上下文：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-事件溯源订单.md
│       └── 0002-写模型使用postgres.md
└── src/
```

如果根目录存在 `CONTEXT-MAP.md`，说明仓库有多个上下文。该文件指向每个上下文所在位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 特定上下文的决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

按需创建文件——只在有内容可写时才创建。如果不存在 `CONTEXT.md`，在第一个术语确定时再创建。如果不存在 `docs/adr/`，在需要第一条 ADR 时再创建。

## 会话过程中

### 与术语表进行比对挑战

当用户使用的术语与 `CONTEXT.md` 中已有定义相冲突时，立即指出。"你的术语表将'取消'定义为 X，但你现在似乎是指 Y——到底是哪个？"

### 打磨模糊语言

当用户使用含糊或重载的术语时，提出一个精确的规范术语。"你说的'账户'——是指客户（Customer）还是用户（User）？这是两个不同的概念。"

### 讨论具体场景

在讨论领域关系时，用具体场景进行压测。构造能探测边界情况的场景，迫使用户对概念边界给出精确说明。

### 与代码交叉验证

当用户描述某件事如何运作时，检查代码是否与之一致。如果发现矛盾，直接点出："你的代码取消整个订单，但你刚才说支持部分取消——哪个是对的？"

### 进化感知

启用前提：只有当项目 `AGENTS.md` 或等价项目级约束明确声明 Agent 具备自主进化能力，并给出进化授权机制时，才执行本节。否则跳过本节，不发起进化申请，不写入记忆，不写入进化日志。

在 grill 会话中，除了打磨领域模型，也要识别是否需要进化 Agent 能力：

- 如果确认了长期有效、可复用、会影响后续开发判断的项目事实，按 `docs/agent/evolution.md` 发起 `[进化申请-知识库]`，未确认前不得修改 `docs/knowledge/`。
- 如果用户多次纠正本 skill 的产物、流程每次都需要大量人工纠偏，或当前 skill 与 SDD 工作流冲突，发起 `[进化申请-技能]`，未确认前不得修改 skill 文件。
- 如果只是用户长期偏好、协作规则、项目长期约定，且不与旧记忆冲突，可按 `docs/agent/evolution.md` 写入 `docs/agent/memory.md`，并记录进化日志。
- `CONTEXT.md` 只记录领域专家有意义的术语、关系、歧义和示例，不记录 skill 执行规则、Agent 协作偏好或一次性任务细节。

### 实时更新 CONTEXT.md

一旦某个术语明确，立即更新 `CONTEXT.md`。不要积攒后再统一写入——随时发生随时记录。格式参见 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)。

不要将 `CONTEXT.md` 与实现细节耦合。只收录对领域专家有意义的术语。

### 谨慎提出 ADR

只有同时满足以下三个条件时，才提议创建 ADR：

1. **难以逆转** —— 之后改变主意的代价是实质性的
2. **缺乏背景会令人困惑** —— 未来的读者会疑惑"为什么这样做？"
3. **真实权衡的结果** —— 存在真实的备选方案，并出于特定原因选择了其中一个

如果三个条件中有任何一个不满足，跳过 ADR。格式参见 [ADR-FORMAT.md](./ADR-FORMAT.md)。
