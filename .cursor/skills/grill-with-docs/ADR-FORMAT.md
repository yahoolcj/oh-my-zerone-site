# ADR 格式

ADR 文件存放在 `docs/adr/` 目录下，使用顺序编号：`0001-slug.md`、`0002-slug.md`，以此类推。

按需创建 `docs/adr/` 目录——只在需要第一条 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么、我们决定了什么、以及为什么这样决定。}
```

就这些。一条 ADR 可以只有一段话。其价值在于记录*做出了*某个决策以及*原因*——而不是填写各种章节。

## 可选章节

只有在真正能提供额外价值时才添加以下内容。大多数 ADR 不需要它们。

- **状态** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）—— 在决策被重新审视时有用
- **已考虑的备选方案** —— 只有当被否决的备选方案值得记忆时才列出
- **后果** —— 只有当非显而易见的下游影响需要特别说明时才列出

## 编号规则

扫描 `docs/adr/` 中已有的最大编号，加一即为新编号。

## 何时提议创建 ADR

以下三个条件必须同时满足：

1. **难以逆转** —— 之后改变主意的代价是实质性的
2. **缺乏背景会令人困惑** —— 未来的读者看到代码会疑惑"为什么要这样做？"
3. **真实权衡的结果** —— 存在真实的备选方案，并出于特定原因选择了其中一个

如果一个决策很容易逆转，跳过它——直接改就行。如果它不令人困惑，没人会疑惑为什么。如果根本没有真实的备选方案，那只是"做了显而易见的事"，没什么可记录的。

### 哪些情况符合条件

- **架构形态。** "我们使用 monorepo。""写模型使用事件溯源，读模型投影到 Postgres。"
- **上下文之间的集成模式。** "订单和账单通过领域事件通信，而非同步 HTTP。"
- **具有锁定效应的技术选型。** 数据库、消息总线、认证提供商、部署目标。不是每个库——只是那些换掉需要花一个季度的。
- **边界与范围决策。** "客户数据由客户上下文拥有；其他上下文只通过 ID 引用。"明确的"不做什么"与"做什么"同样有价值。
- **刻意偏离显而易见路径的决策。** "我们使用手写函数 而非 官方SDK，原因是 X。"任何让合理读者会假设相反做法的地方。这能阻止下一位工程师去"修复"一个刻意为之的设计。
- **代码中不可见的约束。** "由于我们是vue2框架，技术不匹配，我们不能vue3的xx包。""因为当前还没有真实后端接口，所以mock接口并且添加200ms的延迟模拟这是请求。"
- **被否决的备选方案，当否决原因并不显而易见时。** 如果你考虑过 GraphQL 而选择了 REST，且原因比较微妙，就记录下来——否则六个月后会有人再次提出 GraphQL。
