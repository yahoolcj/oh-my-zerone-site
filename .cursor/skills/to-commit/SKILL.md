# Git Commit & PR 生成

## Commit 信息规范

格式：`[type]([module]): 简短描述`

### type 可选值

| type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响逻辑） |
| `refactor` | 重构（非 feat/fix） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建、依赖、工程配置等杂项 |
| `revert` | 回滚 |
| `ci` | CI/CD 配置 |

### 生成步骤

1. 运行 `git diff --staged` 查看暂存区变更（若暂存区为空，运行 `git diff` 查看工作区变更）
2. 识别变更的主要模块（`module`），取组件目录名、页面名或功能模块名，**全小写，不超过 20 字符**
3. 根据变更性质选择 `type`
4. 简短描述控制在 **50 字符以内**，使用中文或英文（与项目一致）
5. 若变更跨多个 module，选最核心的一个，或用 `*` 表示全局

### 示例

```
feat(button): 新增 loading 状态支持
fix(table): 修复空数据时崩溃问题
refactor(form): 将校验逻辑收敛到 useFormValidate
docs(readme): 更新快速上手说明
chore(deps): 升级 vue-demi 到 0.14.x
```

### 异常反馈

- 如果执行add提交命令失败，碰到权限确实或拒绝等原因，将完整提交命令返回给用户，说明原因并引导用户手动提交。

## PR 描述生成

### 生成步骤

1. 询问用户：**"这个 PR 要合并到哪个目标分支？"**（如 `main`、`develop`、`release/x.x.x`）
2. 运行 `git log origin/<目标分支>..HEAD --oneline` 梳理本次 PR 包含的 commit
3. 基于 commit 列表，填写下方模版中的"本次改动说明"和"风险说明"
4. 输出完整 PR 描述

### PR 描述模版

```markdown
## 本次改动说明

<!-- 简要描述本次 PR 的目的、核心改动和影响范围 -->

## 是否使用 AI 辅助

- [x] 是
- [ ] 否

## AI 参与内容

- [ ] 生成页面\功能 / 组件初稿
- [ ] 辅助重构
- [ ] 生成测试用例
- [ ] 辅助排查 Bug
- [ ] 生成文档
- [ ] 其他：

## 开发者自查

- [ ] 我已阅读并理解 AI 生成的代码
- [ ] 我确认代码符合项目规范
- [ ] 我确认没有引入无关依赖
- [ ] 我确认没有删除已有业务逻辑
- [ ] 我确认没有泄露密钥、接口地址、客户数据等敏感信息
- [ ] 我已本地运行并验证功能
- [ ] 我已补充或更新必要测试

## 风险说明

请说明本次改动可能影响的模块或潜在风险。
```

> **注意**："开发者自查"部分的复选框保持未勾选状态，由开发者自行确认后手动勾选。"AI 参与内容"根据本次实际情况勾选对应项。
