# AFK 指数指标


@group:oh-my-zerone


AFK 指数用于衡量一个 PRD 经过 issue 切片后，理论上可以由 AI 独立推进的比例。它不是代码质量指标，也不是工时估算；它只回答一个问题：当前需求拆分里，有多少工作可以在用户离开键盘时持续推进。

## 1. 指标定义

- `AFK`：Agent 可独立完成，用户只需要在最终验收或异常阻塞时介入。
- `HITL`：Human in the loop，执行前、执行中或验收前需要用户明确输入、确认或外部操作。
- `AFK + HITL`：主体工作可由 Agent 完成，但存在一个或多个明确的人类介入点。
- `Manual`：主要依赖用户手动完成，例如必须由用户本机执行脚本、登录第三方平台、完成业务确认或人工验收。

AFK 指数取值范围为 `0` 到 `100`：

- `100` 表示全部 issue 都可以由 Agent 独立推进。
- `0` 表示全部 issue 都需要人类主导。
- 中间值表示人机协作比例，数值越高，越适合异步交给 Agent 执行。

## 2. 输入来源

计算 AFK 指数时只使用 PRD 和 issue 文件中的显式信息：

1. PRD 的功能清单、非目标、验收标准、约束说明。
2. `.ISSUES/<version>-<feature>/README.md` 的 issue 列表、类型和前置依赖。
3. 单个 issue 文件中的 `类型`、`HITL 事项`、`实现范围`、`验收标准`、`后续依赖`。

如果 PRD 和 issue 对同一事项描述不一致，以 issue 文件为准；如果 issue 没有明确标注类型，按 `AFK + HITL` 处理，避免高估自动化比例。

## 3. issue 基础分

每个 issue 先按类型获得一个基础 AFK 分：

| 类型 | 基础分 `B` | 说明 |
|------|-----------:|------|
| `AFK` | 1.00 | Agent 可以按上下文和仓库约定独立完成 |
| `AFK + HITL` | 0.65 | Agent 可完成主体实现，但存在用户确认、外部操作或人工验收 |
| `Manual` | 0.00 | 主要由用户完成，Agent 最多提供说明或检查清单 |

没有明确类型时，`B = 0.65`。

## 4. 工作量权重

不同 issue 对整体工作量的影响不同，需要给每个 issue 一个工作量权重 `W`。优先使用 issue 中的实现范围勾选项数量；没有清单时按 issue 复杂度人工给分。

| 权重 `W` | 适用场景 |
|---------:|----------|
| 1 | 单点修改、文档补充、轻量导出、少量配置 |
| 2 | 单组件内聚实现，包含 3 到 6 个明确任务项 |
| 3 | 跨源码、样式、类型、文档或 demo 的完整切片 |
| 5 | 跨多个组件、影响公共协议、需要复杂兼容或回归验证 |

当 issue 有可勾选的实现范围时，可使用以下规则自动映射：

```text
W = 1, 当任务项数量为 1-2
W = 2, 当任务项数量为 3-6
W = 3, 当任务项数量为 7-12
W = 5, 当任务项数量大于 12 或涉及公共协议变更
```

## 5. 人类介入惩罚

`AFK + HITL` issue 需要根据人类介入强度扣分。对每个 issue 计算 `H`：

| HITL 事项 | 惩罚值 |
|----------|-------:|
| 需要用户确认需求、口径、边界或取舍 | 0.10 |
| 需要用户手动执行脚本、安装依赖、启动服务或提供命令输出 | 0.20 |
| 需要用户登录外部系统、配置密钥、上传文件或完成第三方操作 | 0.30 |
| 需要用户进行主观视觉验收、业务验收或最终发布确认 | 0.15 |
| HITL 事项不明确，只写了 `HITL` 但没有说明 | 0.25 |

单个 issue 的 `H` 上限为 `0.60`，避免一个 issue 被重复扣到没有主体价值。

最终 issue AFK 得分：

```text
S = max(0, B - H)
```

对于 `AFK` issue，通常 `H = 0`。如果 issue 虽标记为 `AFK`，但正文出现必须用户介入的事项，应按实际事项计算 `H`。

## 6. 总公式

设共有 `n` 个 issue：

```text
AFK指数 = round(100 * Σ(Wi * Si) / Σ(Wi))
人类介入比例 = 100 - AFK指数
```

其中：

- `Wi` 是第 `i` 个 issue 的工作量权重。
- `Si` 是第 `i` 个 issue 的最终 AFK 得分。
- `round` 表示四舍五入到整数。

## 7. 分级解释

| AFK 指数 | 解释 | 建议 |
|---------:|------|------|
| 85-100 | 高 AFK | 可以交给 Agent 连续执行，只在验收点回收 |
| 70-84 | 中高 AFK | 适合 Agent 主导，开始前需要补齐少量确认项 |
| 50-69 | 协作型 | 需要分阶段推进，每完成 1 到 2 个 issue 回收一次 |
| 30-49 | 高 HITL | 用户需要频繁参与，应先收敛需求或拆小 issue |
| 0-29 | 人工主导 | 不适合直接 AFK 执行，应先完成外部准备或业务决策 |

## 8. 示例

假设某组件切成 6 个 issue：

| Issue | 类型 | 权重 `W` | HITL 惩罚 `H` | 得分 `S` |
|------|------|---------:|--------------:|---------:|
| 01 初始化目录 | `AFK + HITL` | 3 | 0.20 | 0.45 |
| 02 核心定位 | `AFK` | 2 | 0 | 1.00 |
| 03 数据渲染 | `AFK` | 2 | 0 | 1.00 |
| 04 事件与关闭 | `AFK` | 2 | 0 | 1.00 |
| 05 样式主题 | `AFK` | 2 | 0 | 1.00 |
| 06 文档类型验收 | `AFK + HITL` | 3 | 0.15 | 0.50 |

计算：

```text
Σ(Wi * Si) = 3*0.45 + 2*1 + 2*1 + 2*1 + 2*1 + 3*0.50 = 10.85
Σ(Wi) = 14
AFK指数 = round(100 * 10.85 / 14) = 78
人类介入比例 = 22
```

结论：该需求属于中高 AFK。Agent 可以主导开发，但启动目录创建和最终验收需要用户介入。

## 9. 使用规范

1. 在 `.ISSUES/<version>-<feature>/README.md` 中保留每个 issue 的 `类型` 列。
2. 凡是标记为 `AFK + HITL` 的 issue，必须在单个 issue 文件中写明 `HITL 事项`。
3. 切片时优先把 HITL 点集中到启动、外部资源准备、最终验收等少数 issue，避免每个 issue 都需要用户确认。
4. 如果一个 issue 的 AFK 得分低于 `0.40`，应考虑拆成 `Manual` 准备 issue 和 `AFK` 实现 issue。
5. 最终汇报时同时给出 `AFK指数` 和 `人类介入比例`，不要只给单一数值。
